## המסגרת

פייפליין שמריץ את כל הטסטים בכל commit הוא בזבזני. Impact Analysis מזהה מה בדיוק השתנה ומריץ רק את מה שרלוונטי.

**העיקרון:** ניתוח Git diff מזהה אילו קבצים/מודולים השתנו, ומיפוי dependencies מזהה אילו טסטים/build steps תלויים בהם. השאר, לא צריך להריץ.

**חיסכון פוטנציאלי:** עד 90% מזמני ריצה מיותרים נחסכים כשהפייפליין לא רץ על כל ה-suite בכל פעם.

**מנגנון בסיסי:**
1. ניתוח ה-diff של ה-commit
2. מיפוי מודולים/קבצים שהושפעו
3. Cross-reference עם test coverage map
4. הרצת רק הטסטים הרלוונטיים

**הסיכון המרכזי:** אם מיפוי ה-dependencies לא מדויק, Impact Analysis עלול לדלג על טסט קריטי שהיה אמור לרוץ. תמיד יש להריץ full suite תקופתית (למשל לילי) כ-safety net.

**מדידת TTM (Time-to-Market):** כמה זמן עובר מ-commit ועד deployment. Impact Analysis משפר את המדד הזה משמעותית.

## מה לעשות

**שלב 1:** בנה מנגנון Impact Analysis בסיסי שמנתח Git commits ובוחר שלבי ריצה רלוונטיים בלבד.

**שלב 2:** שלב אותו בפייפליין הקיים שלך (מ-L2-M5).

**שלב 3:** מדוד חיסכון בפועל: זמן ריצה לפני ואחרי. תעד גם safety net (full suite תקופתי).
